The Architecture

Running native binaries on Android is deceptively complex. While Android uses the Linux kernel, its userspace environment is powered by Bionic libc, not the standard Glibc found on desktop Linux.

Most "Moltis on Android" guides force you to install a 1.5GB Ubuntu distribution using proot-distro. This adds massive overhead and latency.

We chose the hard way: Native Execution. No containers, no emulation, just raw performance.

The Solution: Static Musl

To solve the compatibility barrier, we pivoted to an aarch64-unknown-linux-musl target. Unlike standard builds, Musl allows for a fully static binary.

By forcing the C Runtime into the binary itself (target-feature=+crt-static), we produced a standalone executable that carries its own dependencies. It doesn't care about Android's lack of Glibc—it just runs.

Static CRT

No external dynamic linkers required. Truly portable.

Zero Proot

Uses ~20MB of storage instead of 2GB for Ubuntu.

Native Speed

Direct syscalls to the Android kernel. Zero latency overhead.

The "Goated" Tunnel

Development on a phone usually means struggling with SSH keys, port forwarding, and unstable connections. We implemented the VS Code Remote Tunnel (Goat Edition).

Instead of standard SSH, we use the Native VS Code CLI (Alpine/Musl) directly inside Termux. This creates an encrypted bridge that lets you connect your PC browser or VS Code desktop to your phone with one click.

  • No SSH Password prompts.
  • No Port Forwarding required.
  • Full VS Code Desktop performance.
moltis-tunnel - Command to start the secure bridge.

Cloud-Powered CI/CD

Compiling Moltis on a phone can take 30+ minutes and drain the battery. We automated the entire build process using GitHub Actions.

Our automated pipeline:

  • Checks the official Moltis repo every 24 hours.
  • Automatically cross-compiles for the Android target.
  • Injects specific Android patches (PIE fixes, Node-swapping).
  • Pushes a fresh moltis-termux-aarch64.tar.gz release.
